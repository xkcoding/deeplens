## ADDED Requirements

### Requirement: Hub-and-Spoke document generation
The system SHALL invoke `query()` with the "Deep Writer" system prompt and the confirmed outline JSON. For each domain in the outline, the Agent SHALL generate: (1) a Hub document (`index.md`) providing global business perspective and macro data flow diagrams, (2) Spoke documents for each key file/component providing implementation details. Hub documents SHALL reference Spoke documents via Markdown links. Spoke documents SHALL cross-reference each other where logical dependencies exist.

#### Scenario: Generate Hub document for a domain
- **WHEN** the generation agent processes the "User Authentication" domain
- **THEN** it creates `domains/user-auth/index.md` containing a high-level overview, data flow diagram (Mermaid), and links to spoke documents

#### Scenario: Generate Spoke documents for a domain
- **WHEN** the generation agent processes files mapped to "User Authentication"
- **THEN** it creates spoke documents like `domains/user-auth/controller.md` and `domains/user-auth/middleware.md`, each with implementation details and local diagrams

#### Scenario: Cross-references between spokes
- **WHEN** `AuthController` calls `AuthMiddleware`
- **THEN** the controller's spoke document contains a link to the middleware's spoke document

### Requirement: Mermaid diagram generation
The Agent SHALL generate Mermaid.js diagram code within Markdown files. Hub documents SHALL contain macro-level diagrams (sequence diagrams or flowcharts showing cross-module data flow). Spoke documents SHALL contain local diagrams (component interactions within a single module). Diagrams SHALL use business concept names, NOT raw class/function names (e.g., "User Strategy" instead of "AbstractUserStrategyImpl").

#### Scenario: Hub document contains sequence diagram
- **WHEN** generating a hub document for "Order Processing"
- **THEN** the document contains a Mermaid sequence diagram showing the data flow from order creation through payment to fulfillment

#### Scenario: Smart simplification in diagrams
- **WHEN** generating a diagram for a module that contains logging, metrics, and business logic
- **THEN** the diagram only shows business logic interactions, omitting Logger, Metrics, and ConnectionPool

### Requirement: Noise filtering in document content
The Agent SHALL apply "Smart Simplification" strategy when generating document content. It SHALL focus on core business logic and data flow. It SHALL omit: logging calls, metrics instrumentation, connection pool management, generic exception handling boilerplate, and standard framework annotations/decorators. It SHALL explicitly state when content has been simplified (e.g., "Error handling and logging omitted for clarity").

#### Scenario: Business logic extraction
- **WHEN** generating documentation for a service that contains 200 lines including 50 lines of logging and error handling
- **THEN** the document focuses on the ~150 lines of business logic, noting that error handling was omitted

### Requirement: Architecture overview document
The system SHALL generate a top-level `architecture.md` document in the docs root. This document SHALL contain: project summary, detected technology stack, a high-level system architecture diagram (Mermaid), and a table of all domains with brief descriptions and links.

#### Scenario: Architecture overview generated
- **WHEN** all domain documents are complete
- **THEN** `architecture.md` exists with a complete system overview, architecture diagram, and domain index table

### Requirement: Index page generation
The system SHALL generate an `index.md` document as the VitePress home page. It SHALL contain: project name, one-paragraph summary, quick navigation to key domains, and a "Generated by DeepLens" attribution.

#### Scenario: Home page links to domains
- **WHEN** the documentation site is opened
- **THEN** the index page shows the project name, summary, and clickable links to each domain's hub document

### Requirement: Iterative generation with progress tracking
The Agent SHALL process domains sequentially (or in a prioritized order). After completing each domain, the system SHALL log progress (e.g., "Completed 3/7 domains: Order Processing"). The system SHALL save intermediate results — if the process is interrupted, completed domain documents SHALL be preserved.

#### Scenario: Progress reporting
- **WHEN** the Agent finishes generating documents for the 3rd of 7 domains
- **THEN** the CLI displays "✓ Order Processing (3/7 domains complete)"

#### Scenario: Interrupted generation preserves work
- **WHEN** the generation process is interrupted (e.g., Ctrl+C) after completing 4 of 7 domains
- **THEN** the 4 completed domain directories and their documents are preserved on disk
