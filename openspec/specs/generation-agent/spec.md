## MODIFIED Requirements

### Requirement: Index page generation
The system SHALL generate an `index.md` document as the VitePress home page from the outline's `overview` data. The `index.md` SHALL contain: project name as H1 title, `overview.architecture` rendered as a section with Mermaid diagram, `overview.tech_stack_roles` as a table (columns: Technology, Role), `overview.key_flows` as a numbered description list, `overview.project_structure` as a descriptive section, navigation links to each domain's hub document, a Quick Start section with 2-3 actionable next steps for new developers linking to specific domain docs, and a "Generated by DeepLens" attribution.

#### Scenario: Home page generated from overview
- **WHEN** the generation agent processes a confirmed outline with overview data
- **THEN** `index.md` contains architecture section, tech stack table, key flows, structure description, domain navigation links, and Quick Start section

#### Scenario: Home page links to domains
- **WHEN** the documentation site is opened
- **THEN** the index page shows the project name, architecture overview, and clickable links to each domain's hub document

#### Scenario: Quick Start section
- **WHEN** the overview generator completes index.md
- **THEN** the Quick Start section provides 2-3 concrete, actionable steps linking to specific domain docs

### Requirement: Architecture overview document
The system SHALL NOT generate a separate top-level `architecture.md` document. All architecture content (project summary, detected technology stack, system architecture diagram, domain index table) SHALL be consolidated into `index.md` via the overview data. This avoids content duplication between `architecture.md` and `index.md`.

#### Scenario: No standalone architecture.md
- **WHEN** all domain documents are complete
- **THEN** no `architecture.md` file exists in the docs root; architecture content is in `index.md`

### Requirement: Summary page generation
The system SHALL generate a `summary.md` document as the project summary / wrap-up page AFTER the overview is generated. The `summary.md` SHALL be produced by an independent `runSummaryGenerator()` sub-agent that reads the overview and all domain hub documents. The `summary.md` SHALL contain: "Project Summary" as H1 title, At a Glance table (key metrics), Domain Recap (1-2 sentences per domain), Cross-cutting Concerns (patterns spanning domains), Architectural Insights (global perspective), Potential Improvements (tech debt, gaps), and "Generated by DeepLens" attribution.

#### Scenario: Summary generated after overview
- **WHEN** the overview generator completes index.md
- **THEN** the summary generator runs and produces `summary.md`

#### Scenario: Summary content is evidence-based
- **WHEN** the summary generator reads domain docs
- **THEN** the Domain Recap section references actual content from domain hub documents

#### Scenario: Summary phase progress
- **WHEN** the summary generator starts
- **THEN** a progress event with `phase: "summary"` and `completed: 0, total: 1` is emitted
- **AND WHEN** summary.md is written
- **THEN** a progress event with `phase: "summary"` and `completed: 1, total: 1` is emitted
